# ТЗ: сервисные вкладки «Обслуживание» + полный экран готовки (session `wiva-android-service-tabs`)

> **Область:** `wiva-android`. **Поведение:** `wiva_electron`. **Оболочка сервисного меню и паттерны полей:** `legacy Android kiosk` (см. основное ТЗ §2). **Родительский документ:** `docs/TZ_WIVA_ANDROID_FROM_ELECTRON.md`. **Конфигурация сборки/тестов:** `AGENTS.md` в корне `wiva-android`.

---

## Описание задачи

Необходимо заменить заглушки `WivaPlaceholderTab` для трёх подвкладок группы «Обслуживание» и довести пользовательский экран готовки напитка до согласованного с эталоном уровня. Заглушки и навигация уже заданы в `WivaServiceMenuTabContent.kt`; структура групп/табов — в `WivaServiceMenuStructure.kt` (группа `Maintenance`: Остатки, Калибровка сиропов, Калибровка воды).

**Инварианты из основного ТЗ**

- Новые сценарии настроек и обслуживания не выносятся в отдельные полноэкранные обходные маршруты — только существующие ячейки сервисного меню (§2, таблица «Сервисное меню»).
- Семантика протокола контроллера и обменов, соответствующая калибровке и готовке, берётся из **wiva_electron**; транспорт/обвязка на Android — по существующему контракту приложения (без дублирования смысла эталона в этом документе).
- Детали реализации не копируются в ТЗ — ниже только ссылки на эталонные пути и проверяемые результаты.

**Четыре логических модуля приёмки**

| Модуль | Суть |
|--------|------|
| M1 — CalibrationWater | Таб «Калибровка воды»: тестовый налив, ввод измеренного объёма, сохранение коэффициента и производных данных (скорость налива и т.д., как в эталоне). |
| M2 — CalibrationSyrup | Таб «Калибровка сиропов»: выбор контейнера, тестовый налив, ввод результата, сохранение — по потоку эталона. |
| M3 — PreparingProcess | Экран готовки: визуальный и сценарный паритет с эталоном (видеослой, прогресс, успех, модалка чека — по эталону); расчёт оценочного времени с учётом калибровки воды. |
| M4 — Inventory | Таб «Остатки»: отображение данных об остатках в духе эталона; списание при готовке уже выполняется в сервисе готовки — в scope отображения и согласованности данных. |

## Юзер-кейсы и сценарии

### Актёры

- **Сервисный инженер / администратор** — входит в сервисное меню, выполняет калибровку и просмотр остатков.
- **Покупатель** — видит экран готовки после оплаты и выбора напитка (клиентский сценарий).

### M1 — CalibrationWater

**Эталоны поведения и UX:** `wiva_electron/src/renderer/components/ServiceMenu/tabs/WaterCalibrationTab/` и модальная часть `wiva_electron/src/renderer/components/ServiceMenu/tabs/CalibrationTab/WaterCalibrationModal/`.

**Основной сценарий**

1. Пользователь открывает «Обслуживание» → «Калибровка воды».
2. Видит текущие сведения о калибровке (целевой/фактический объём, коэффициент, при наличии — скорость налива и длительность последнего налива), согласованные с эталоном по смыслу.
3. Задаёт целевой объём тестового налива в допустимых пределах и инициирует калибровочный налив.
4. Во время налива отображается состояние ожидания/прогресса; по завершении налива (успех по контроллеру) пользователь вводит фактически измеренный объём.
5. Пользователь подтверждает расчёт и запись коэффициента; система сохраняет данные так, чтобы последующие расчёты времени готовки могли использовать скорость налива, **как предусмотрено эталоном** (см. связку с M3).

**Альтернативные и ошибочные сценарии**

- Тайм-аут или ошибка налива: пользователь получает понятное сообщение; повторная попытка возможна без «залипания» UI (логика соответствует эталону main/renderer).
- Невалидные значения объёма: ввод отклоняется с сообщением; запись коэффициента не выполняется при отсутствии необходимых данных о длительности/результате налива, если эталон так предписывает.

**Связь с IPC/событиями в эталоне (семантика, не перенос строк кода):** каналы и потоки `WATER_CALIBRATION_POUR`, `WATER_CALIBRATION_WRITE_COEFFICIENT`, `GET_WATER_CALIBRATION_INFO`, событие результата налива (в electron — `WATER_CALIBRATION_POUR_RESULT` / см. `wiva_electron/src/common/ipc/controller.ts` и модуль контроллера). На Android реализуется эквивалентная цепочка вызовов к контроллеру и хранению настроек без обязательного совпадения имён классов.

---

### M2 — CalibrationSyrup

**Эталоны:** UI и шаги — `wiva_electron/src/renderer/components/ServiceMenu/tabs/CalibrationTab/`; состояние и действия — `wiva_electron/src/renderer/state/calibration/`.

**Основной сценарий**

1. Пользователь открывает «Обслуживание» → «Калибровка сиропов».
2. Выбирает контейнер из списка, согласованного с конфигурацией автомата (как в эталоне).
3. Запускает тестовый налив для выбранного контейнера.
4. После завершения налива вводит измеренный результат (объём или величина по смыслу эталона).
5. Сохраняет калибровку; отображается подтверждение или обновлённые параметры.

**Альтернативные сценарии**

- Смена контейнера до сохранения: поведение как в эталоне (сброс или сохранение черновика — по эталону, не по догадке).
- Ошибка контроллера или тайм-аут: сообщение пользователю и возможность повторить.

---

### M3 — PreparingProcess (полный экран готовки)

**Эталоны UI и потока экрана:** `wiva_electron/src/renderer/pages/Preparing/` (включая видеослой, кольцо прогресса, экран успеха, модалку чека — перечень компонентов смотреть в репозитории эталона).

**Эталон расчёта времени/объёма для таймера:** `wiva_electron/src/main/services/drink/DrinkSelectionService.ts` и связанные сценарии **PreparingManager** (как в electron).

**Текущее состояние Android:** `app/src/main/java/com/wiva/android/ui/screens/customer/PreparingScreen.kt` — упрощённый аналог; навигация с параметрами — `WivaNavGraph.kt` (маршрут с `productId`, `productName`, `estSeconds`).

**Основной сценарий (покупатель)**

1. После успешной оплаты и выбора напитка пользователь попадает на экран готовки с оценочным временем, согласованным с расчётом из калибровки воды (скорость налива из M1 при наличии; иначе — поведение как в эталоне для значения по умолчанию).
2. На экране отображаются элементы эталона: фон/видеослой, индикация прогресса, тексты состояний.
3. По завершении готовки показывается экран/секция успеха как в эталоне.
4. При необходимости отображается модалка чека/чека — по сценарию эталона.

**Связка с бэкендом приложения:** сервис готовки `WivaDrinkPreparingService.kt` должен оставаться согласованным с цепочкой «оплата → команда контроллеру»; расчёт `estSeconds` при переходе на экран — с использованием актуального **flow rate** из калибровки воды (логика из `DrinkSelectionService.ts` / эталона, не дублируется здесь).

**Альтернативные сценарии**

- Прерывание или ошибка готовки: отображение состояния ошибки или возврат — по эталону и существующим ответам контроллера.
- Отсутствие данных калибровки: используется дефолтная скорость, как зафиксировано в эталоне документацией/кодом electron.

---

### M4 — Inventory (остатки)

**Эталон UI:** `wiva_electron/src/renderer/components/ServiceMenu/tabs/InventoryTab/`.

**Данные:** остатки приходят из телеметрии и локальной агрегации (репозиторий/таблица — как в существующем коде Android для «Наполнения»); **списание** при успешной готовке выполняется в `WivaDrinkPreparingService.kt` — в задаче M4 фокус на **отображении** и актуальности данных для вкладки «Остатки», а не на дублировании логики списания.

**Основной сценарий**

1. Пользователь открывает «Обслуживание» → «Остатки».
2. Видит таблицу или список позиций в терминах эталона (контейнеры, объёмы, при необходимости пороги — по `InventoryTab`).
3. Может обновить данные вручную, если эталон предусматривает кнопку обновления.

**Альтернативные сценарии**

- Нет связи с телеметрией или пустые данные: понятное состояние UI (как в эталоне или согласованно с паттерном `legacy Android kiosk` для пустых таблиц).

**Разграничение с табом «Телеметрия → Наполнение»:** в `WivaServiceMenuStructure.kt` это разные подвкладки; смысл каждой должен соответствовать своему эталону в electron (`InventoryTab` vs матрица/merge в телеметрии — не смешивать назначение экранов).

## Нефункциональные требования

- **Сборка и тесты:** после изменений выполняется `buildCommand` и при необходимости unit-тесты из `AGENTS.md`.
- **Единообразие сервисного UI:** шапка, rail, `ScrollableTabRow`, колонки полей — как в `ServiceScreen.kt` / `ServiceTabUtils.kt` и образцах `legacy Android kiosk` (`ServiceMenuScreen`, `tabs/ServiceTabUtils.kt`).
- **Доступность на эмуляторе:** сценарии калибровки и готовки должны быть проверяемы с **моком контроллера** (основное ТЗ §3), с теми же ожидаемыми последовательностями событий, что и для реального устройства.
- **Производительность UI:** экран готовки не блокирует главный поток; видео/анимации — без заметных артефактов на целевом API (`AGENTS.md`: эмулятор `wiva`, API 25).
- **Вопросы владельцу:** при неоднозначности, влияющей на поведение или приоритет, и отсутствии ответа в ТЗ/эталонах — остановка и запрос по формату §4.1 основного ТЗ (не вопросы про реализацию).

## Критерии приёмки

### M1 — CalibrationWater

- [ ] В `WivaServiceMenuTabContent.kt` для `WivaServiceSubTabId.CalibrationWater` отображается реализованный контент, а не заглушка.
- [ ] Пользователь может выполнить полный цикл: задать объём → тестовый налив → ввести измеренный объём → сохранить; отображаются актуальные параметры калибровки до/после, по смыслу как в эталоне `WaterCalibrationTab` / `WaterCalibrationModal`.
- [ ] Ошибки и тайм-ауты налива обрабатываются без зависания; сообщения соответствуют ожиданиям эталона.
- [ ] Сохранённые данные доступны для расчёта времени готовки (связь с M3) и при необходимости читаются тем же способом, что заложен в приложении (аналог `GET_WATER_CALIBRATION_INFO` по смыслу).

### M2 — CalibrationSyrup

- [ ] Для `WivaServiceSubTabId.CalibrationSyrup` — полноценный таб вместо заглушки.
- [ ] Реализованы выбор контейнера, тестовый налив, ввод результата и сохранение; порядок шагов и сообщения согласованы с `CalibrationTab` и state `wiva_electron/src/renderer/state/calibration/`.
- [ ] Регрессия: остальные подвкладки сервисного меню открываются без ошибок.

### M3 — PreparingProcess

- [ ] Экран готовки включает элементы, перечисленные в эталоне `wiva_electron/.../pages/Preparing/` (видеослой, прогресс, success-view, receipt-modal — по наличию в эталоне; медиа при необходимости согласовать с политикой копирования ассетов в `AGENTS.md`).
- [ ] `estSeconds` (или эквивалент для анимации) согласован с расчётом из объёма напитка и **flow rate** калибровки воды по правилам `DrinkSelectionService.ts`; при отсутствии калибровки — поведение как в эталоне.
- [ ] `WivaDrinkPreparingService.kt` и навигация к экрану согласованы: нет расхождения между временем на UI и командами контроллеру.
- [ ] Ручная проверка на эмуляторе: happy-path «выбор напитка → оплата (мок) → экран готовки → завершение» проходит.

### M4 — Inventory

- [ ] Для `WivaServiceSubTabId.Inventory` — реализованный таб вместо заглушки.
- [ ] Данные остатков соответствуют источнику телеметрии/репозитория; после готовки напитка отображаемые остатки согласованы со списанием в `WivaDrinkPreparingService.kt` (интеграционно или ручной сценарий).
- [ ] UI и терминология соответствуют эталону `InventoryTab` по смыслу (колонки, подписи, действия).

### Общий DoD сессии

- [ ] `WivaServiceMenuStructure.kt` не требует изменения порядка группы «Обслуживание» — только замена содержимого ячеек.
- [ ] Новые composable/VM для табов размещены в `.../service/tabs/` по конвенции проекта (`AGENTS.md`).
- [ ] Сборка успешна; критические линтер-замечания по изменённым файлам устранены.
