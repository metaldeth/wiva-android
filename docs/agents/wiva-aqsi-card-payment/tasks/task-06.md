## Что сделать

- Расширить **`WivaServiceMenuStructure.kt`**: новая группа «Оплата картой» и три подвкладки (Метод; aQsi — Настройки; aQsi — Диагностика) — **только добавление**, существующие группы/вкладки не менять.
- Расширить **`WivaServiceMenuTabContent.kt`** (или роутер эквивалента): `when` по id → новые composable.

  - **Метод:** выбор PAX vs AQSI, сохранение через `CardPaymentMethodRepository`.  
  - **Настройки:** host, port, timeout; сохранить → `AqsiRepository.saveConfig`; «Тест соединения» → **`testTcpConnection`** (не устаревший `connect()` из черновика request). ViewModel: лог **`AqsiSettings`** при сохранении и результате теста.  
  - **Диагностика:** текущий метод карты; сводка последней операции из **in-memory** state (**holder** из **task-03**/**task-04**), обновляемого при завершениях `testTcpConnection`, платежей/отмен из **task-03** и **любом** исходе AQSI-платежа с экрана заказа из **task-05**; тест платежа 1 коп. только под **`BuildConfig`/dev/mock** guard.

### Чеклист UC-5: согласованность с заказом

- После сценария AQSI с **экрана заказа** (напиток или подписка, моки допустимы): открыть вкладку **Диагностика** — отображаются **актуальные** метки времени/исхода (approve / decline / error / cancel), согласованные с последним завершением `initiatePayment`, без сырых payload.
- После **только** теста TCP или теста 1 коп. из сервисного меню сводка также отражает последнюю операцию (приоритет последней по времени — по принятой в реализации политике; зафиксировать в MR при неоднозначности).
- Диагностический «лог последней транзакции» — только агрегированный исход, без сырых payload.

## Тест-кейсы (TDD / unit)

### ViewModel вкладки «Метод»

1. Загрузка: при репозитории с AQSI UI-state отражает AQSI.
2. Выбор PAX и сохранение: вызван `setSelected(PAX)`.

### ViewModel «Настройки» (`AqsiSettings`-логика тестируется через поведение; логи — ручной чек в task-07)

3. **Сохранить:** вызывается `saveConfig` с введёнными полями (порт/таймаут валидируются по UX правилам экрана).
4. **Тест соединения — успех:** при `Result.success` показывается успешное состояние UI.
5. **Тест соединения — ошибка:** при ошибке TCP UI показывает ошибку без краша.

### ViewModel «Диагностика»

6. Отображение текущего `CardPaymentMethod` из репозитория.
7. **Сводка после последней операции:** при предзаполненном holder (симуляция: установить snapshot в тесте) UI-state отражает исход **approve / decline / error / cancel / tcp-test** и краткий код без утечки payload.
8. **Согласованность с заказом (интеграционный unit / фейки):** после вызова фейкового завершения платежа «как с заказа» (через публичный API holder или через фейковый репозиторий, вызываемый так же, как из оркестратора) вкладка-ViewModel читает те же поля, что обновил **task-05** / **task-03**.
9. **Guard теста 1 коп.:** в release / без dev guard кнопка недоступна или no-op (assert на флаг).
