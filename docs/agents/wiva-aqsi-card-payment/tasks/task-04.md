# task-04 — DI: AqsiModule

**Зависимости:** task-03

## Связь с юзер-кейсом

Обеспечивает внедрение `AqsiRepository`, клиента и связанных зависимостей в приложение без правок интеграционного модуля.

## Что сделать

- Создать `di/AqsiModule.kt`: `@Module`, `@InstallIn(SingletonComponent::class)`.
- Связать `AqsiRepository` → `AqsiRepositoryImpl`, при необходимости провайдинг `Arcus2Client` (singleton или фабрика — по принятому стилю).
- Все новые биндинги aQsi — **только** в этом модуле.
- В этом же модуле (или связанном `@Module` в том же файле) предоставить **singleton** для **in-memory** сводки последней операции aQsi: holder/тип **объявлены в task-03**, здесь — **`@Provides @Singleton`** (или `@Binds` к реализации), чтобы **AqsiRepositoryImpl**, **CardPaymentOrchestrator** (если нужен доступ) и экран диагностики получали **один** экземпляр.

## Точки изменения

- `app/src/main/java/com/wiva/android/di/AqsiModule.kt` (новый)

**Не трогать:** `IntegrationsModule.kt`.

## Тест-кейсы (TDD — **обязательны**, не сводить задачу к одной сборке)

Задача **не считается выполненной**, если единственная проверка — успешный `assembleDebug`. Нужны **явные** автотесты валидности DI-слоя aQsi.

1. **Компиляция графа Hilt (обязательно):** успешные `gradlew.bat assembleDebug` (или как минимум `:app:compileDebugJavaWithJavac` / kapt Hilt без ошибок — но **не заменяет** пункты 2–3).
2. **Smoke резолва графа (обязательно, предпочтительно instrumented):** минимальный **`@HiltAndroidTest`** в `src/androidTest` (или принятый в проекте эквивалент), который поднимает тестовое приложение с Hilt и **достаёт** из графа хотя бы **`AqsiRepository`** (например через `@Inject`, `EntryPointAccessors` или тестовый `Activity` `@AndroidEntryPoint`). Допускается подмена тяжёлых зависимостей через `@TestInstallIn`, `UninstallModules` + тестовый модуль с фейками — цель **именно** «граф собирается и бинды резолвятся», а не E2E оплаты.
3. **Контракт провайдера / fallback (обязательно, если instrumented невозможен в CI):** если по политике репозитория **нельзя** добавить `androidTest` в этом MR — зафиксировать в MR причину и вместо п.2 выполнить **два** обязательных минимума:
   - **3a.** Unit-тест (Robolectric **или** чистый JVM — по возможности), который создаёт **узкий тестовый компонент** / `Dagger` builder **только** для подмодуля aQsi (если принят паттерн) и проверяет, что `@Provides`/`@Binds` методы `AqsiModule` отдают ожидаемые типы; **или**
   - **3b.** Отдельный **`MainDispatcher`/`@Module` test** контракт: тест, что сгенерированный Hilt-код содержит привязки `AqsiRepository` ↔ `AqsiRepositoryImpl` (например через инспекцию сгенерированных классов **только если** это уже есть в проекте — не изобретать хрупкие рефлексии).

   Без п.2 **или** согласованной пары **3a+обоснование** задача **не закрывается**.

4. **Holder диагностики (unit, по возможности):** если в модуле участвует абстракция `AqsiDiagnosticsSnapshotHolder`/эквивалент — один unit-тест, что она **singleton** в рамках тестового модуля / что `@Provides` возвращает тот же экземпляр при повторном запросе (если применимо к выбранному стилю).

*Инструментальные тесты:* следовать существующей конфигурации `wiva-android` (зависимости `hilt-android-testing`, правила пакетов). Если в проекте уже есть пример `@HiltAndroidTest` — скопировать паттерн.
