## Инвариант DrinkListViewModel (архитектурное ревью)

Оба карточных входа должны идти через **`CardPaymentOrchestrator`**:

1. **`paidTerminalThenPour`** — только при **`sbp == false`** (чистая карта); при **`sbp == true`** код **не менять**, оркестратор карты **не** вызывать (**UC-1b / G3**).
2. **`startSubscriptionPayment(isSbp = false)`** — заменить прямой `sendSumToTerminal` на тот же оркестратор с параметрами, эквивалентными текущему вызову; при **`isSbp == true`** — без изменений.

После успеха PAX или AQSI — **тот же** пост-успех (налив / подписка / телеметрия), что и сегодня после успешной карты.

## Альтернативные ветки UC-1 / UC-2 (aQsi и регрессия PAX)

Основные шаги UC покрыты задачами из таблицы ниже. **Содержательные** альтернативы из ТЗ разложены по чеклистам задач (не откладывать на «наследие приложения» без явных критериев).

| Сценарий | UC | Где в плане | Критерий уровня реализации / проверки |
|----------|-----|-------------|----------------------------------------|
| Отказ терминала (decline) | UC-1 PAX | **task-05** | Поведение как **до** внедрения оркестратора: те же сообщения / возврат в безопасное состояние заказа; **нет** налива и **нет** успешной телеметрии при неуспехе. Регрессия по моку контроллера / чеклист MR. |
| Таймаут / отмена (PAX) | UC-1 | **task-05** | Как в текущем приложении; оркестратор **не** меняет контракт `PaymentTerminalService`; пути отмены/таймаута сохраняются. |
| Отказ ридера (decline по протоколу) | UC-2 aQsi | **task-03** (парсер + маппинг), **task-05** | `AqsiPaymentResult.Declined` → `CardPaymentResult` неуспех; UI/VM — безопасное состояние, **без** налива и **без** успешной телеметрии. |
| Таймаут канала / операции | UC-2 | **task-03** | Таймаут из `AqsiConfig.timeoutMs`; маппинг в ошибку без uncaught exception; unit-покрытие см. task-03. |
| Обрыв сокета | UC-2 | **task-03** | IOException / broken pipe → контролируемая ошибка, лог `AqsiRepo`; см. тест-кейсы task-03. |
| Некорректный / усечённый пакет (invalid packet) | UC-2 | **task-03** | Ошиб protocol parse → `Error`, без падения процесса; unit на фикстурах. |
| Отмена пользователем / оператором | UC-2 | **task-03**, **task-05** | `cancelPayment()` и согласование с жизненным циклом платежа в VM; итог как отмена / неуспех **без** пост-успеха. |
| Безопасное состояние UI заказа | UC-1, UC-2 | **task-05** | После любого неуспеха/отмены карты: **нет** запуска налива и **нет** отправки **успешной** sale-телеметрии; сообщения согласовать с существующими ошибками оплаты где уместно. |
| In-memory диагностика после **реальной** AQSI-оплаты с экрана заказа | UC-5 | **task-03** (сеансы репозитория: платёж, тест TCP), **task-05** (поток заказа), **task-06** (отображение) | Сводка последней операции обновляется при **любом** завершении AQSI-платежа с заказа (approve/decline/error/cancel); вкладка диагностики показывает актуальные поля; см. задачи. |
