# Архитектура: aQsi Pill / T7100 (оплата картой, Arcus2 TCP/IP)

**SessionId:** `wiva-aqsi-card-payment`  
**Проект:** `wiva-android` (`type=android`)  
**Источники:** утверждённое `tz.md`, `request.md`, `AGENTS.md`.

## Границы и инварианты

- **Не затрагиваются (без изменений кода):** `hardware/controller/*`, `PaymentTerminalService.kt`, `domain/model/PaymentMethod.kt`, `di/IntegrationsModule.kt`, существующие вкладки Equipment (Платёжник, Контроллер), Integrations > СБП.
- **Вариативность «карта»:** только отдельная доменная модель `CardPaymentMethod` (PAX / AQSI) и JsonStore; enum приложения `PaymentMethod` (NONE / SBP / CARD) не расширяется.
- **Транспорт в объёме MR:** Arcus2 поверх **TCP/IP** к `host`/`port` (типично USB-Ethernet). **VCOM/COM без TCP-туннеля** — вынесено за пределы текущей задачи, если в кодовой базе нет готового паттерна; допускается отдельный ADR/эпик после появления паттерна или явного ТЗ.
- **DI:** все новые привязки aQsi — только **`AqsiModule.kt`** (`@InstallIn(SingletonComponent::class)`), без правок других Hilt-модулей интеграций.
- **Персистентность:** только **новые** ключи в `JsonStoreKeys` + сериализация значений; существующие ключи и строковые значения не меняются.
- **Контрольные сборки (Windows):** обязательный успешный `gradlew.bat assembleDebug` после этапов **A2, A4, A5, A6**; финальная сборка по завершении всего объёма.

## Интерфейсы и контракты

### JsonStore (новые ключи)

| Ключ константы | Строковое значение | Содержимое |
|----------------|-------------------|------------|
| `CARD_PAYMENT_METHOD` | `"cardPaymentMethod"` | Строка `"PAX"` \| `"AQSI"` (дефолт при отсутствии — PAX). |
| `AQSI_SETTINGS` | `"aqsiSettings"` | JSON объекта `AqsiConfig` (host, port, timeoutMs). |

### Персистентность диагностики aQsi и границы данных

Чтобы не хранить и не распространять чувствимые платёжные данные, зафиксировано следующее.

| Категория | Где хранится | Что допустимо |
|-----------|----------------|---------------|
| Выбор метода карты | **JsonStore**, ключ `CARD_PAYMENT_METHOD` | Только `"PAX"` / `"AQSI"`. |
| Конфиг TCP | **JsonStore**, ключ `AQSI_SETTINGS` | `host`, `port`, `timeoutMs` — неплатёжные сетевые параметры. |
| Сводка «последний тест соединения» / «последняя операция» для экрана диагностики | По умолчанию **только память процесса** (singleton / state holder, обновляется из `AqsiRepository` и при необходимости из оркестратора): timestamp, исход (OK / ошибка / approve / decline), **краткий** человекочитаемый код или enum-статус, длительность/фаза **без** сырого тела пакетов. После холодного старта экран диагностики может быть пустым по «последней операции» — это согласованный компромисс PCI/безопасности; при явном продуктовом требовании «после перезапуска» — **отдельный новый** ключ JsonStore **только** с такими же нечувствительными meta-полями (без PAN/трека/CVV/полных payload). |
| Логи Timber | Не хранилище, но политика | См. теги; байты Arcus2 — только debug; никогда не логировать PAN, трек, CVV, полные криптограммы. |

**Никогда не персистить** (и не кэшировать в JsonStore/DataStore произвольно): PAN, трек-2/подобное, CVV/CVC, полные сырые кадры JPAY/Arcus2 с платёжным полезным грузом, PIN-блоки, данные магнитной полосы. Для UI «лог последней транзакции» — только **агрегированный** исход и **безопасные** коды ответа протокола/драйвера, не восстанавливающие чувствительный контент.

Дополнительные ключи JsonStore для диагностики **не обязательны**; если вводятся — только по таблице выше и с отдельным согласованием в MR.

### `CardPaymentMethodRepository`

- `suspend fun getSelected(): CardPaymentMethod`
- `suspend fun setSelected(method: CardPaymentMethod)`

### `AqsiRepository`

- `suspend fun loadConfig(): AqsiConfig` (с дефолтами полей при пустом хранилище)
- `suspend fun saveConfig(config: AqsiConfig)`
- `suspend fun testTcpConnection(): Result<Unit>` (или `Boolean` + message) — **проверка канала без финансовой транзакции:** TCP connect к `host`/`port` в пределах `timeoutMs`, graceful close. Если по документации/поведению T7100 для валидности канала недостаточно одного TCP connect, допускается **минимальный** обмен уровня handshake/idle probe, **не** инициирующий JPAY-платёж (не `INIT_PAYMENT` на реальную сумму). Это **безопаснее и реалистичнее**, чем «тест» через полноценную операцию списания. Устаревшая формулировка в `request.md` про `AqsiRepository.connect()` трактуется как этот метод/операция на репозитории, без отдельного публичного `connect()` в домене, если не понадобится иное при реализации.
- `suspend fun initiatePayment(amountKopecks: Int): Result<AqsiPaymentResult>`
- `suspend fun cancelPayment(): Result<Unit>`

Сигнатуры могут использовать `Result` на уровне домена; внутри data — перехват IOException и таймаутов.

### `CardPaymentOrchestrator`

- Метод уровня заказа: принимает те же **бизнес-параметры**, что сейчас уходят в `sendSumToTerminal` в **каждом** из двух мест вызова (напиток и подписка) — сумма / номер продукта / тип продукта в терминах приложения (конкретные имена задаются при реализации); контракт: **не дублировать** логику 0x48 внутри оркестратора, а **передавать** в `PaymentTerminalService` уже готовые аргументы, как сегодня в соответствующих ветках ViewModel.
- `suspend fun pay(…): CardPaymentResult` (или эквивалент; при необходимости — перегрузка или единый параметр-носитель, покрывающий оба сценария без копипасты бизнес-правил).
- Отмена: `fun cancel()` / `suspend fun cancel()` — согласованно с текущим жизненным циклом платежа в ViewModel.

## Модели данных

### `AqsiConfig`

- `host: String`
- `port: Int` (default 16107)
- `timeoutMs: Long` (default 15000)

Представление в JsonStore — один JSON на ключ `AQSI_SETTINGS`.

### Результаты

- **`AqsiPaymentResult`:** `Approved`, `Declined(reason: String)`, `Error(cause: Throwable или типизированный код)` — без утечки PAN/CVV в `message`.
- **`CardPaymentResult`:** `Success`, `Failed(reason)`, `Cancelled` — маппинг из PAX (существующие статусы терминала) и AQSI.

### Телеметрия продажи

- При **Approved** в ветке AQSI вызывается тот же путь, что и после успешной карточной оплаты PAX; в JSON оплаты поле метода остаётся **`CARD`** (без новой строки для ридера в этом MR).

## Интеграции и потоки

### Потоки и диспетчеры

- Любые `Socket` connect/read/write, парсинг Arcus2 — строго вне main: **`Dispatchers.IO`** (или `withContext(IO)` из use case/репозитория).
- UI (Compose) — только сбор состояния и вызов `viewModelScope.launch` / suspend из ViewModel.

### UC-1: Оплата картой, выбран PAX

1. Пользователь выбирает оплату картой на экране заказа.
2. `DrinkListViewModel` → `CardPaymentOrchestrator.pay(...)` (**оба** сценария: напиток через `paidTerminalThenPour` при **`sbp == false`**, и подписка при **`startSubscriptionPayment(isSbp = false)`**).
3. Оркестратор читает `CardPaymentMethod` = PAX → вызывает **нес изменений** `PaymentTerminalService.sendSumToTerminal(...)`.
4. Успех → существующий путь налива + телеметрия (как сегодня).

### UC-1b: `paidTerminalThenPour` — карта и СБП (инвариант G3)

Внутри **`paidTerminalThenPour`** вызов **`CardPaymentOrchestrator.pay(...)`** выполняется **только** если активна **чисто карточная** ветка терминала (**`sbp == false`**), т.е. тот же путь, где сегодня вызывается `sendSumToTerminal` для карты. Если **`sbp == true`** (сценарий, где после СБП снова задействуется терминал по текущим правилам приложения), логика **остаётся прежней** — **без** подмены на оркестратор карты и **без** изменения последовательности вызовов СБП/терминала. Так фиксируется требование: СБП **неизменен**, глобальный переключатель **CardPaymentMethod** влияет только на истинные CARD-платежи.

### UC-2: Оплата картой, выбран AQSI

1. Тот же вход с экрана заказа (напиток или подписка — см. UC-1), **не** смешивая с СБП.
2. Оркестратор читает AQSI → `AqsiRepository.initiatePayment(amountKopecks)`.
3. Репозиторий на IO: `Arcus2Client` открывает TCP, выполняет сценарий JPAY (init / ожидание ответа / при необходимости cancel), закрывает ресурсы.
4. `Approved` → **тот же** post-success путь, что у PAX (налив + телеметрия, `method = CARD`).
5. Отказ / ошибка → единый неуспех карточной оплаты; UI не падает; логи по тегам.

### UC-3–UC-5: Сервисное меню

- **Метод:** запись в `CARD_PAYMENT_METHOD`.
- **Настройки:** редактирование полей → `saveConfig` → `AQSI_SETTINGS`; «Тест соединения» → `testTcpConnection` (TCP reachability / минимальный handshake **без** JPAY-транзакции; см. контракт `AqsiRepository`).
- **Диагностика:** чтение текущего метода из репозитория; отображение последней известной операции из in-memory state (обновляется репозиторием/оркестратором); «Тест 1 коп.» за guard `BuildConfig` / dev / mock.

### Отмена

- При активном PAX — делегировать в существующий механизм отмены терминала (как в текущем приложении), **без** изменения `PaymentTerminalService`.
- При активном AQSI — `AqsiRepository.cancelPayment()` / сигнал клиенту на cancel-пакет протокола.

### Транспорт VCOM (отложено)

- Если в проекте появится общий абстракция «поток байт ↔ Arcus2», VCOM может реализовать тот же контракт, что и TCP; **в текущем MR** проектируется и реализуется только TCP/IP.

## Риски и точки внимания при реализации

- Согласовать сигнатуру `CardPaymentOrchestrator.pay` с фактическими параметрами **обоих** путей: `paidTerminalThenPour` (карта, `sbp == false`) и `startSubscriptionPayment(isSbp == false)` / `sendSumToTerminal`, не размножая бизнес-логику.
- Таймауты: использовать **только** `AqsiConfig.timeoutMs` на соединение/операции, не дублировать магические константы в ViewModel.
- R8/release: новые Hilt-компоненты попадают под существующие правила проекта; при добавлении сериализации для `AqsiConfig` — keep rules при необходимости (отдельная проверка `assembleRelease` вне минимального DoD ТЗ, но полезно перед релизом).
