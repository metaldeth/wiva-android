# Архитектурное ревью: `wiva-aqsi-card-payment` (круг 2)

**SessionId:** `wiva-aqsi-card-payment`  
**Проект:** `wiva-android`  
**Объект:** `architecture.md` в сопоставлении с `tz.md`, `request.md`, `AGENTS.md` и замечаниями круга 1.

---

## Краткий вывод

Документ **`architecture.md`** на круге 2 **закрывает критичный пробел круга 1**: глобальный `CardPaymentMethod` (PAX / AQSI) распространяется на **обе** карточные точки в `DrinkListViewModel` — **`paidTerminalThenPour`** (при **`sbp == false`**) и **`startSubscriptionPayment(isSbp == false)`**, с явным запретом трогать ветку **`isSbp == true`**. Добавлен явный инвариант **UC-1b / G3**: при **`sbp == true`** оркестратор карты **не** подмешивается в последовательность СБП/терминала.

Описаны контракт **`testTcpConnection`** (вместо устаревшего `connect()` из черновика `request.md`), таблица тегов Timber согласована с **A7** из `tz.md`, политика PCI/диагностики (in-memory по умолчанию) уточнена.

**Блокирующих противоречий между `architecture.md` и `tz.md` не выявлено.** Остаточный риск — **рассинхрон с `request.md`**, который всё ещё в ряде мест описывает только сценарий напитка и устаревшие детали; для реализации и приёмки источником правды остаются **`tz.md` + `architecture.md`** (как уже зафиксировано в архитектуре).

## Рекомендации / замечания

### Синхронизация `request.md` с утверждённой архитектурой (важно для команды)

- В **`request.md`** в блоке изменений `DrinkListViewModel` по-прежнему фигурирует в основном **`paidTerminalThenPour`**; для снижения риска регресса **желательно** явно добавить **`startSubscriptionPayment`** (карта, `isSbp == false`) и условие **`!sbp`** для оркестратора в `paidTerminalThenPour`.
- В **жёстких правилах** `request.md` встречается пример тега **`AqsiPayment`**; **приёмка A7** и **`tz.md`** ориентируются на **`CardPayment`**, **`AqsiRepo`**, **`Arcus2`**, **`AqsiSettings`**. Имеет смысл **привести `request.md` к тем же тегам**, чтобы не плодить расхождения с logcat-чеклистом.

### Декомпозиция в `tz.md`

- Краткая декомпозиция в **`tz.md`** по-прежнему формулируется вокруг одной замены в ViewModel; это **не противоречит** уточнённому `architecture.md`, но для единого onboarding можно **одной фразой** дополнить ТЗ вторым входом (подписка) — по желанию владельца постановки.

### Унификация `CardPaymentOrchestrator.pay(...)`

- В разделе **рисков** `architecture.md` корректно подчёркнута необходимость **одной** сигнатуры/носителя параметров для **обоих** вызовов без дублирования бизнес-правил — это остаётся главным местом внимания на этапе **A5**.

### Прочее (наследие круга 1, без изменения статуса)

- **Диагностика UC-5:** компромисс «после перезапуска пусто» vs опциональный безопасный ключ — уже зафиксирован в архитектуре; отдельного блокирующего замечания нет.
- **Release / R8:** по **`AGENTS.md`** полезна проверка **`installRelease`** на AVD `wiva` — вне минимального DoD `tz.md`, но уместна в чеклист MR.
