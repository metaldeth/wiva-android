# Техническое задание: интеграция aQsi Pill / T7100 (оплата картой)

**SessionId:** `wiva-aqsi-card-payment`  
**Проект:** `wiva-android`  
**Источник постановки:** `docs/agents/wiva-aqsi-card-payment/request.md`

---

## Описание задачи

### Цель

Добавить в приложение wiva-android альтернативный путь **оплаты банковской картой** через устройство **aQsi Pill / T7100**: взаимодействие по протоколу **Arcus2** в режиме **TCP/IP к хосту и порту** (типично поверх USB-Ethernet по документации KB) и прикладному уровню **JPAY**, **без участия контроллера автомата** для этого канала.

**Объём транспорта в этой задаче:** реализация ориентирована на сценарий **Arcus2 TCP/IP** (`AqsiConfig`: host, port, timeout). Путь **VCOM/COM без TCP-туннеля** в объём задачи **не входит**, если в кодовой базе нет уже готового локального паттерна или заказчик не задаёт явное отдельное требование на COM — такие случаи выносятся за пределы текущего MR.

Текущий путь **PAX через контроллер** (команды 0x48 / ответ 0x56 со статусом «успех = 4») сохраняется без изменений в реализации `PaymentTerminalService` и без изменений в слое контроллера.

Переключение между способами оплаты **картой** (PAX vs aQsi) — **единственное** затрагиваемое изменение методологии платежей; **СБП**, **freeMode** и общий enum `PaymentMethod` (NONE / SBP / CARD) **не изменяются**.

### Жёсткие ограничения (нарушение = блокирующий дефект)

1. **Контроллер только для чтения:** не изменять `hardware/controller/*`, в том числе `RequestCommand.kt`, `ResponseCommand.kt`, `ControllerGateway.kt`; новые команды контроллеру не добавлять.
2. **PAX-слой:** не изменять `services/payment/PaymentTerminalService.kt`; вызывать его из нового оркестратора «как есть» для ветки PAX.
3. **СБП:** не изменять `startSbpPayment()`, репозитории и UI вкладки СБП.
4. **Существующие сервисные вкладки:** не менять UI и логику вкладок `Equipment > Платёжник`, `Equipment > Контроллер`, `Integrations > СБП`; допускается только **добавление** новой группы/вкладок для карточной оплаты и aQsi.
5. **Доменный enum способов оплаты приложения:** не изменять `domain/model/PaymentMethod.kt`; вариативность реализуется отдельной моделью **только для карты** (например, sealed `CardPaymentMethod`: PAX / AQSI).
6. **Хранилище:** в `JsonStoreKeys` добавлять **только новые** ключи; существующие значения ключей не менять.
7. **DI:** новый модуль `AqsiModule.kt`; **не изменять** `IntegrationsModule.kt`.
8. **Логирование:** Timber с отдельными тегами для новых классов по постановке (например `CardPayment`, `AqsiRepo`, `AqsiSettings`, для клиента протокола — `Arcus2` по постановке); не менять теги существующих классов (например `PaymentTerminal`). Логирование **сырых байт** протокола Arcus2 (`Arcus2`) — **только в debug-сборках**; не логировать чувствительные платёжные данные (PAN, трек-данные, CVV и аналоги).
9. **Телеметрия успешной оплаты через aQsi:** вызывать тот же путь отправки продажи/демо-телеметрии, что и при успешной оплате в PAX-ветке; в JSON оплаты поле вида «method» для карты остаётся **`CARD`**, без введения нового типа строки для ридера в рамках этой задачи (отдельное различение — вне объёма, по постановке).

Если вынос во вкладочный подпакет технически конфликтует с уже принятой раскладкой `wiva-android`, допускается размещение в `ui/screens/service/` с явным обоснованием в MR — по умолчанию ориентир **`tabs/`**.

### Краткая декомпозиция (для последующих подзадач разработки)

- **Модель выбора способа оплаты картой:** отдельный sealed-тип (PAX / aQsi), ключ `cardPaymentMethod` в JsonStore со строковыми значениями вида `"PAX"` / `"AQSI"`, значение по умолчанию PAX; репозиторий/use case чтения-записи.
- **Домен aQsi:** `AqsiConfig` (host, port по умолчанию 16107, timeout по умолчанию 15000 ms), интерфейс `AqsiRepository`, модель результатов платежа (одобрено / отказ / ошибка), ключ хранилища для JSON конфигурации.
- **Данные:** `Arcus2Client` (connect, отправка транзакции/отмена, получение результата, disconnect); `AqsiRepositoryImpl`; unit-тест парсинга ответных пакетов на fixture-номерных данных.
- **DI:** модуль `AqsiModule` без правок интеграционного модуля.
- **Оркестрация карты:** `CardPaymentOrchestrator` выбирает ветку по сохранённому `CardPaymentMethod`: PAX — **только делегирование** в существующий **`PaymentTerminalService.sendSumToTerminal(...)`** без изменений класса и без изменений PAX/контроллерской логики; aQsi — `AqsiRepository.initiatePayment` по сумме в копейках; отмена в зависимости от активного метода; унифицированный результат (успех / неуспех / отмена). В **`DrinkListViewModel`** заменить в сценарии карты прямой вызов `sendSumToTerminal` на вызов оркестратора; ветки SBP, mock controller, free mode не трогать; **после успешной оплаты** (PAX или aQsi) выполняется **тот же** неизменённый код налива и телеметрии, что и сегодня после успешной карточной оплаты.
- **Сервисное меню:** новая группа «Оплата картой» с вкладками: выбор метода (PAX / aQsi), настройки aQsi (host, port, timeout, сохранить, тест соединения), диагностика (статус, лог последней операции; тестовый платёж на 1 коп — только как описано в постановке для dev/mock). UI новых вкладок — в пакете `tabs/` по **`AGENTS.md`**, если структура проекта это поддерживает (см. § «По структуре проекта»).
- Порядок внедрения по постановке: A1 → A2 → A3 → A4 → A5 → A6 → A7. **Обязательные контрольные сборки** (Windows): успешный **`gradlew.bat assembleDebug`** после **A2, A4, A5, A6** — как в `request.md`. Сборки после A1, A3, A7 — по необходимости при коммите, без требования «после каждого» этапа. **Финальная** успешная **`gradlew.bat assembleDebug`** по завершении всей задачи **обязательна** перед закрытием MR.

## Юзер-кейсы и сценарии

### Актёры

- **Покупатель** — экран заказа, выбирает оплату картой.
- **Обслуживающий персонал / инженер** — сервисное меню настройки и диагностики.
- **Системы** — контроллер + PAX (без изменения протокола), ридер aQsi по TCP, локальное хранилище настроек, телеметрия.

### UC-1. Оплата картой при выбранном методе PAX

1. В настройках выбран метод карты «PAX» (или значение по умолчанию).
2. Покупатель завершает заказ оплатой картой (как сейчас).
3. Приложение обращается к оркестратору карты; тот вызывает **существующий** `PaymentTerminalService.sendSumToTerminal(...)` **как есть** (PAX-логика и протокол контроллера 0x48/0x56 **не меняются** и **не дублируются** в оркестраторе).
4. Успешный платёж ведёт к **тому же** сценарию налива и телеметрии, что при текущей карточной оплате через PAX (без отдельной ветки «только для PAX» после одобрения).

**Альтернативы:** отказ терминала, таймаут, отмена — как в текущем приложении (без указания переопределений в этом ТЗ).

### UC-2. Оплата картой при выбранном методе aQsi

1. В сервисном меню выбран метод «aQsi Pill», задан корректный `AqsiConfig`.
2. Покупатель инициирует оплату картой.
3. Оркестратор не использует команду контроллера 0x48 для этого пути; выполняется сессия с ридером по Arcus2/JPAY; результат трактуется как одобрено / отказ / ошибка связи или протокола.
4. При **одобрении** выполняется **тот же** кодовый путь, что после успешной карточной оплаты через PAX: налив, демо/прод телеметрия в том же виде (включая `method = CARD` в JSON оплаты), **без** ветвления «если aQsi — отдельный пост-успех».
5. При отказе или ошибке операция заканчивается как неуспешная оплата картой; приложение не падает, ошибки логируются указанными тегами.

**Альтернативы:** неверный host/port; таймаут; обрыв сокета; отмена пользователем/оператором — отображение и возврат в безопасное состояние UI заказа (детали UX совместить с существующими сообщениями об ошибках платежей, где уместно).

### UC-3. Выбор метода оплаты картой в сервисном меню

1. Инженер открывает группу «Оплата картой», вкладка выбора метода.
2. Доступен выбор между PAX (через контроллер) и aQsi (контроллер для карты не задействован).
3. Выбор сохраняется и восстанавливается после перезапуска приложения.

### UC-4. Настройка и проверка aQsi

1. Инженер на вкладке настроек aQsi задаёт Host, Port, Timeout, сохраняет.
2. По кнопке «Тест соединения» выполняется проверка возможности установить канал связи по конфигурации с понятным результатом успех/ошибка (конкретный API операции может быть методом репозитория или операцией клиента — определяется на этапе проектирования API без смягчения требования «проверка TCP к ридеру»).

### UC-5. Диагностика aQsi

1. На вкладке диагностики отображаются: текущий выбранный метод карты, сведения о последнем подключении/сеансе (по постановке), краткий лог последней транзакции (approve/decline/error).
2. «Тест платежа» на 1 копейку выполняется **только** в описанном в постановке ограниченном режиме (например dev/mock), чтобы не провоцировать лишних списаний на продакшен-конфигурации.

## Нефункциональные требования (если есть)

- **Android API:** не ниже `minSdk`, зафиксированного в `app/build.gradle.kts` (для этого проекта актуально API 25, см. AGENTS.md).
- **Потоки:** сеть и блокирующие I/O операции Arcus2 — не на главном потоке (`IO`/аналог проекта).
- **Тайм-ауты:** использовать настраиваемый `timeoutMs` из конфигурации aQsi, не захардкоживать постоянными магическими числами в бизнес-логике.
- **Надёжность:** ошибки сокета, таймауты, некорректные пакеты — без необработанных исключений на UI; результат через `Result` или эквивалент; журналирование.
- **Arcus2 / PCI-гигиена логов:** подробный (в т.ч. байтовый) лог обмена по `Arcus2` — только **debug**; в **release** не допускать избыточного лога с потенциально чувствимыми платёжными данными.
- **Персистентность настроек:** по образцу существующих JSON/DataStore ключей проекта (как для SBP-настроек), только новые ключи.
- **Тестируемость:** unit-тест на разбор типового ответного пакета Arcus2 (одобрено / отказ) на фикстурах после реализации парсера.
- **Изоляция задачи:** ни один файл из явного запрета постановки не изменять; один MR на весь объём по ветке `dev`.
