# Декомпозиция: viwa-android-from-electron

Основана на `wiva-android/docs/TZ_WIVA_ANDROID_FROM_ELECTRON.md` §4 и §6.2. Каждый подмодуль = отдельный complex, артефакты в `docs/agents/viwa-android-from-electron/modules/{moduleId}/`.

## Модули

### Модуль 1: `stage-a-carcass-ota`

- **Описание:** Gradle-приложение (Kotlin, Compose), базовая навигация, dev-флаг мока контроллера, OTA-сервер и клиент как в legacy Android kiosk (контракт `version.json`, документация по образцу `legacy Android kiosk/docs/OTA_UPDATE.md`).
- **Скоуп:** `wiva-android/` (новый app-модуль, `update-server/`, `docs/`).
- **Зависимости:** нет (первый этап).
- **ID задач ТЗ:** A1–A4.

### Модуль 2: `stage-b-mock-first`

- **Описание:** Реализация **MockController** по контракту, инъекция вместо реального транспорта; сценарий на эмуляторе (команда → ответ в логе/UI). Смысл — из `wiva_electron` моки (`MockControllerConnection.ts` и тесты).
- **Скоуп:** слой контроллера в `wiva-android`, DI.
- **Зависимости:** `stage-a-carcass-ota` (каркас, флаг мока).
- **ID задач ТЗ:** B3.

### Модуль 3: `stage-b-real-controller`

- **Описание:** Serial/USB под Android, байтовый протокол 1:1 с wiva, связка оплата/терминал → контроллер как в `PaymentTerminalService` и вызывающем коде wiva.
- **Скоуп:** транспорт, парсер, платёжная цепочка.
- **Зависимости:** `stage-b-mock-first` (контракт и переключение реал/мок).
- **ID задач ТЗ:** B1, B2, B4.

### Модуль 4: `stage-c-integrations`

- **Описание:** MAX, СБП, Нанокасса из `legacy Android kiosk` (data/domain, DI); C1–C3 **не** в одном PR без промежуточных сборок.
- **Скоуп:** `data/remote/*`, репозитории, настройки.
- **Зависимости:** `stage-a-carcass-ota` минимум; удобнее после наличия базового app и DI-скелета.
- **ID задач ТЗ:** C1–C4.

### Модуль 5: `stage-d-telemetry`

- **Описание:** Транспорт телеметрии из legacy Android kiosk; семантика обменов из wiva_electron; инвентаризация D2; порциями кластеры D3; связка событий D4.
- **Скоуп:** `telemetry`, интеграция с оплатой/подписками.
- **Зависимости:** транспорт и DI из предыдущих этапов; для D4 — потоки оплаты из B/C.
- **ID задач ТЗ:** D1–D4.

### Модуль 6: `stage-e-ui-flow`

- **Описание:** UI и бизнес-флоу автомата (выбор напитка/объём, сервисное меню, остальные экраны — отдельными задачами).
- **Сервисное меню (инвариант):** см. ТЗ §2 таблица «Сервисное меню» — порядок групп/подвкладок = `wiva_electron` `ServiceMenu/constants.ts` (зеркало `ViwaServiceMenuStructure.kt`); оболочка и поля = образец `legacy Android kiosk` `ServiceMenuScreen` + `tabs/`. Любые новые настройки только внутри этой иерархии; тяжёлые вкладки выносить в `app/.../service/tabs/*.kt` по аналогии с киоском.
- **Скоуп:** Compose-экраны, состояния как у wiva renderer где нет конфликта с киоском.
- **Зависимости:** мок контроллера, телеметрия и платежи по мере готовности.
- **ID задач ТЗ:** E1–E3+.

### Модуль 7: `stage-f-office`

- **Описание:** Физический контроллер, калибровка портов, регрессия против мока; реальный платёжный терминал на стенде.
- **Скоуп:** железо, чеклисты, подписи.
- **Зависимости:** все предыдущие модули в рабочем состоянии.
- **ID задач ТЗ:** F1–F2.

## Волны выполнения

| Волна | Модули | Комментарий |
| ----- | ------ | ----------- |
| 1 | `stage-a-carcass-ota` | Старт проекта; после A — желателен минимальный `wiva-android/AGENTS.md` с `buildCommand` (ТЗ §7). |
| 2 | `stage-b-mock-first` | Разблокирует UI на эмуляторе. |
| 3 | `stage-b-real-controller` | Железо и протокол. |
| 4 | `stage-c-integrations` | C1→сборка→C2→сборка→C3→C4 (ТЗ). |
| 5 | `stage-d-telemetry` | D2 желательно рано; кластеры D3 по одному. |
| 6 | `stage-e-ui-flow` | Экраны отдельными задачами в plan complex; сервисное меню — строго по ТЗ §2 (electron-порядок + kiosk-вёрстка). |
| 7 | `stage-f-office` | Офис/стенд, не блокирует мок-разработку. |

Параллельность между волнами **не** задана ТЗ — порядок строго последовательный; внутри волны 4 возможны параллельные задачи только если план модуля это допускает.

## Общие сущности

- **Источники правды:** ТЗ §2 — legacy Android kiosk (MAX, СБП, Нанокасса, транспорт телеметрии, OTA); wiva_electron (протокол контроллера, семантика телеметрии, платежи/терминал).
- **Контракты между модулями:** `ControllerGateway` / `MachineController` (мок + реал); единый `TelemetryManager`-аналог по семантике wiva.
- **Конфигурация сборки:** `applicationId`, flavor, `ANDROID_UPDATE_BASE_URL` — уточнить у владельца (ТЗ §7).
