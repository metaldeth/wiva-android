# План: `stage-b-mock-first` (B3 — MockController / шлюз)

**Сессия:** `wiva-android-from-electron/modules/stage-b-mock-first`  
**ТЗ:** [tz.md](./tz.md)  
**Архитектура:** файл `architecture.md` отсутствует; ниже — вывод из ТЗ и кода `wiva-android`.

## Вывод архитектуры (кратко)

- **Контракт:** единый интерфейс исходящих команд + доставка входящих ответов подписчикам (аналог `sendCommand` + `ControllerEvents` / `simulateResponse` в wiva).
- **Две реализации:** мок по смыслу `MockControllerConnection.ts`; «реал» на B3 — заглушка без B1/B2.
- **Шлюз:** `@Singleton` делегирующий объект: на **каждой** исходящей операции (и при подписке на события при необходимости) читает `ConfigRepository.get(JsonStoreKeys.USE_MOCK_CONTROLLER)` и направляет вызов в мок или в заглушку — **без cold restart**.
- **DI:** Hilt-модуль предоставляет в приложение именно **шлюз** как точку входа; мок и заглушка — внутренние зависимости шлюза или `@Inject` конструкторы + `@Binds`/`@Provides`.
- **Команды/ответы:** типы согласованы с эталоном wiva (`RequestCommand` / `ResponseCommand` в TypeScript) — в Kotlin: `enum`/числовые коды 1:1 по значениям на этом этапе (полный парсер кадров — B2).
- **Проверка UC-3:** согласованная тестовая точка (кнопка на экране сервиса, dev-действие, или unit-тест) + логи; минимальная UI-индикация допустима.
