# Архитектура: `stage-b-mock-first` (B3 — MockController / шлюз)

**Проект:** `wiva-android`, пакет `com.wiva.android` (Android, Kotlin, Compose, Hilt).  
**ТЗ:** `docs/agents/wiva-android-from-electron/modules/stage-b-mock-first/tz.md`.

---

## Компоненты и ответственность

| Компонент | Ответственность |
|-----------|-----------------|
| **Kotlin enums `RequestCommand` / `ResponseCommand`** | Зеркалирование кодов команд и ответов с эталоном Electron: те же числовые значения и семантика имён, что в TS-перечислениях (см. пути ниже). Единый словарь для исходящих TX и входящих RX событий. |
| **`ControllerGateway` (интерфейс)** | Публичный контракт для слоя приложения: отправка команды с телом (байты); подписка на входящие ответы; опционально — точка расширения в духе «mock port» для будущих сценариев наливов (флаг/свойство без жёсткой привязки к B4). |
| **`MockControllerGateway`** | Реализация по смыслу `MockControllerConnection`: журнал исходящих команд; явная симуляция входящего ответа (аналог `simulateResponse`); эмиссия в общий поток ответов так же, как при реальном парсере. |
| **`StubRealControllerGateway`** | Заглушка «реала» до B1/B2: без USB/serial и полного протокола; удовлетворяет контракту (no-op или лог), не падает при выключенном моке. |
| **`SelectingControllerGateway` или `DelegatingControllerGateway`** | Единственный `@Singleton` публичный шлюз в DI: на **каждой** исходящей операции (и при необходимости при настройке подписок) читает флаг мока из `ConfigRepository` по ключу `JsonStoreKeys.USE_MOCK_CONTROLLER` и делегирует в `MockControllerGateway` или `StubRealControllerGateway`. Переключение без перезапуска процесса. |
| **`ConfigRepository`** | Уже существующий репозиторий конфигурации; источник истины для `"true"` / `"false"` строки мока (тот же ключ, что экран сервиса). |
| **Hilt-модуль** | Предоставление трёх реализаций в графе и **привязка `@Singleton` к типу `ControllerGateway`** как к выбирающему/делегирующему шлюзу; мок и заглушка — отдельные `@Singleton` или внутренние зависимости фасада (решение на этапе реализации — без дублирования публичного API). |
| **`ServiceScreen` / `ServiceViewModel`** | Dev-приёмка: кнопка или метод VM для отправки тестовой команды и вызова симуляции ответа; логирование через Timber по шагам сценария UC-3. |

## Интерфейсы и контракты

### Эталон команд в Electron (для зеркалирования enum)

- Исходящие: `wiva_electron/src/main/hardware/controller/commands/RequestCommands.ts` — `export enum RequestCommand`.
- Входящие: `wiva_electron/src/main/hardware/controller/commands/ResponseCommands.ts` — `export enum ResponseCommand`.

### Эталон поведения мока

- `wiva_electron/src/main/hardware/controller/__mocks__/MockControllerConnection.ts` — журнал `sentCommands`, `sendCommand`, `simulateResponse` через `events.emit`.

### Публичный контракт Android (`ControllerGateway`)

Структурно (без кода):

- Метод отправки: принимает `RequestCommand` и тело как байтовый буфер (например `ByteArray` или `UByteArray` — выровнять с остальным проектом).
- Поток входящих событий: **hot** `SharedFlow` (или эквивалент с тем же контрактом доставки многим подписчикам), элементы описывают пару «тип ответа (`ResponseCommand`) + полезная нагрузка».
- Для мока — отдельный контрактный метод симуляции ответа (доступен из тестов и dev-UI; в проде вызывать только осознанно / ограничить build-type при необходимости — вне скоупа данного документа, но точка входа одна).
- Опционально: флаг уровня «mock port» для будущей интеграции с наливами (как `isMockPort` в эталоне) — объявить в мок-реализации или в расширении интерфейса без блокировки B3.

### Выбирающий шлюз

- Зависимости: `ConfigRepository`, две делегируемые реализации, `CoroutineScope` для чтения настроек без блокировки main thread там, где это уместно (например отложенный `suspend`/`withContext` внутри методов).
- Инвариант: решение «мок vs реал» принимается **на каждом вызове** публичного метода отправки (и при инициализации подписки на поток — если потоки у мока и стаба должны быть раздельными или мультиплексированы; предпочтительно единый `SharedFlow` у фасада, собирающий события от активной ветки).

### DI

- `@Singleton` на классе, реализующем `ControllerGateway` и видимом потребителям.
- `@Binds` / `@Provides` в модуле Hilt (например рядом с `AppModule` или выделенный `ControllerModule` под пакет `di`).

## Модели данных

| Сущность | Назначение |
|----------|------------|
| **`RequestCommand`** | Enum: числовой код TX, совместимый с протоколом (источник правды — TS enum). |
| **`ResponseCommand`** | Enum: числовой код RX. |
| **Запись журнала мока** | Пара «команда + тело» для проверок и отладки (аналог элементов `sentCommands`). |
| **Событие входящего ответа** | Упаковка `ResponseCommand` + байтовое тело для эмиссии в `SharedFlow`. |
| **Конфиг** | Строковое значение по ключу `JsonStoreKeys.USE_MOCK_CONTROLLER` (`"true"` / `"false"`). |

Пакет размещения: логично `com.wiva.android.hardware.controller` или `com.wiva.android.domain.controller` для enum и шлюза; `data` не обязателен для чистых enum, если они не тянут платформу.

## Интеграции и потоки

1. **Экран сервиса → `ConfigRepository`**  
   Уже сохраняет `USE_MOCK_CONTROLLER`. Без изменения ключа; шлюз только читает.

2. **Потребитель фичи → `ControllerGateway` (singleton)**  
   Вызов отправки → фасад читает флаг → делегирование в мок или стаб → при моке запись в журнал и (опционально) отсутствие автоматического ответа до `simulateResponse`.

3. **Входящие ответы**  
   Мок: `simulateResponse` → эмиссия в `SharedFlow`. Стаб: либо тишина, либо только лог; без ложных событий.  
   UI/ViewModel подписывается на `SharedFlow` (с учётом `repeatOnLifecycle` / `viewModelScope` по правилам проекта).

4. **Dev-сценарий UC-3**  
   `ServiceViewModel` получает `ControllerGateway` через Hilt; по кнопке на `ServiceScreen` или по методу VM: отправка согласованной тестовой `RequestCommand` + вызов симуляции известного `ResponseCommand` → **Timber** логи: факт отправки, факт симуляции, при необходимости — получение в коллекторе подписчика.

5. **Потоки и Main**  
   Эмиссии в `SharedFlow` не должны нарушать правила главного потока для UI; тяжёлые операции чтения конфига — через корутины и подходящий диспетчер.

## Рекомендации по параллелизации

| Поток работ | Можно параллельно |
|-------------|-------------------|
| **A. Enum `RequestCommand` / `ResponseCommand`** | Независимо: копирование значений из двух TS-файлов в Kotlin (один PR или два, без конфликтов с UI). |
| **B. `MockControllerGateway` + журнал + `SharedFlow` + `simulateResponse`** | Независимо от стаба реала и от Hilt, если контракт `ControllerGateway` зафиксирован текстом/интерфейсом в том же коммите-основе. |
| **C. `StubRealControllerGateway`** | Параллельно с B; единственное согласование — сигнатуры интерфейса. |
| **D. `Selecting`/`Delegating` фасад + Hilt-модуль** | После или вместе с черновиком интерфейса; требует готовых B и C как зависимостей. |
| **E. Интеграция в `ServiceScreen` / `ServiceViewModel` + Timber** | После D; может идти параллельно с юнит-тестами на мок. |
| **F. Юнит-тесты мока** | Параллельно с E; опираются на B. |

Критический путь: интерфейс → (B ∥ C) → D → E/F.
