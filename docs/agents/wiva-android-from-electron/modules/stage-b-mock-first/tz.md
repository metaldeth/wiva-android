# ТЗ: модуль `stage-b-mock-first` — задача B3 (MockController)

**Проект:** `wiva-android` (Android, Kotlin, Compose, Hilt).  
**Источник границ этапа:** `wiva-android/docs/TZ_WIVA_ANDROID_FROM_ELECTRON.md` — таблица модуля B, строка **B3**; §3 «Обязательный мок контроллера».  
**Эталон поведения мока (смысл, не построчный перенос):** `wiva_electron/src/main/hardware/controller/__mocks__/MockControllerConnection.ts` и связанные тесты в том же каталоге.  
**Контекст UI:** переключатель «Мок контроллера» уже сохраняется в локальном хранилище конфигурации (ключ `useMockController` / `JsonStoreKeys.USE_MOCK_CONTROLLER`) и отображается на экране сервиса; задача B3 — связать это значение с выбором реализации на уровне DI/рантайма.

**Вне скоупа B3:** реальный serial/USB-транспорт и полный байтовый протокол (B1, B2); цепочка «оплата → контроллер» (B4). Для ветки «реальный контроллер» на этом этапе допускается **заглушка** (no-op / лог «не подключено»), главное — единый контракт и переключение с моком.

---

## Описание задачи

Ввести **единый контракт** доступа к контроллеру (шлюз / транспорт — терминология из §3 общего ТЗ: например `ControllerGateway` или `MachineController`), с **двумя реализациями**:

1. **Мок** — переносит **смысл** эталона `MockControllerConnection`: фиксация исходящих команд с полезной нагрузкой; возможность **симулировать входящий ответ** контроллера так, чтобы подписчики/слой выше получали те же события, что и при реальном потоке байтов (аналог `simulateResponse` + диспетчеризация по типу ответа). При необходимости для будущего сценария наливов предусмотреть признак поведения в духе `isMockPort` (в эталоне при `true` сервис наливов использует таймеры вместо реальной отправки команды) — если на момент реализации соответствующий use case уже существует в приложении; иначе заложить в контракт/мок точку расширения без блокировки B3.

2. **Реальная реализация на B3** — **не обязана** говорить с железом: достаточно заглушки, реализующей контракт (например логирование или пустые операции), пока этапы B1/B2 не дадут настоящий транспорт.

**Переключение (зафиксировано для B3):** источник истины — то же хранилище, что пишет экран сервиса (`USE_MOCK_CONTROLLER`). Единый `@Singleton` **шлюз** читает флаг из `ConfigRepository` **на каждой исходящей операции** (и при необходимости при подписке на события) и **делегирует** в мок или в заглушку «реала». Так переключатель в сервисе меняет режим **без перезапуска** процесса и без противоречия UC-1. Альтернатива «только со следующего cold start» на этом этапе **не используется**.

Итог B3: на **эмуляторе** можно пройти проверочный сценарий из DoD общего ТЗ: **отправка тестовой команды** через единую точку входа → **ожидаемый ответ мока** виден в логе и/или в UI (минимальная индикация допустима, отдельный полноценный продуктовый экран не требуется).

## Юзер-кейсы и сценарии

**Актёры:** разработчик или тестировщик; приложение в режиме сервисного меню и в обычном режиме использования (когда появится вызов контракта).

| ID | Юзер-кейс | Основной сценарий | Альтернативы / исключения |
|----|-----------|-------------------|---------------------------|
| UC-1 | Включить режим мока | Пользователь открывает экран сервиса, включает «Мок контроллера», значение сохраняется. При следующем обращении приложения к шлюзу контроллера используется мок-реализация. | Флаг уже включён — повторное включение не ломает состояние. |
| UC-2 | Выключить режим мока | Пользователь выключает переключатель; сохраняется «реал» (заглушка). Вызовы идут в не-мок реализацию без падения приложения. | На эмуляторе без железа «реал» может только логировать — это ожидаемо на B3. |
| UC-3 | Проверка обмена с моком | Через согласованную тестовую точку (кнопка/действие dev, unit/integration-тест или временный вызов из существующего кода) отправляется известная **исходящая** команда. В логе видно факт отправки; при симуляции ответа моком — видно **входящее** событие того же типа, что ожидает слой подписок (аналог `events.emit` в эталоне). | Если симуляция не вызвана — подписчики не получают ложный ответ; порядок шагов документируется в `summary.md` этапа. |
| UC-4 | Согласованность с эталоном wiva | Сценарии записи команд и выдачи ответов соответствуют **роли** мока в wiva (журнал `sentCommands`, `simulateResponse`), без обязательного совпадения имён классов и внутренних структур. | Отличия допустимы, если зафиксированы в summary с обоснованием (Kotlin-идиомы, корутины). |

## Нефункциональные требования

- **Сборка:** успешное выполнение команды сборки из `wiva-android/AGENTS.md` (`assembleDebug`; на Windows — `gradlew.bat assembleDebug`) после изменений.
- **DI:** Hilt, область жизни объектов согласована с тем, что флаг мока может читаться при старте и/или обновляться из UI; нет утечек контекста Activity в синглтоны без необходимости.
- **Потокобезопасность / корутины:** если события с мока доставляются асинхронно, не нарушать правила Main thread для UI; избегать гонок при записи журнала команд при параллельных вызовах (или явно ограничить контракт однопоточным использованием с документированием).
- **Зависимости:** мок и заглушка реала не тянут за собой реализацию B1/B2 (USB/serial, полный парсер), если она ещё не введена.
- **Логирование:** достаточно для приёмки B3 — по шагам видно «команда ушла в мок» и «симулирован ответ …» (уровни и теги — по принятому в проекте стилю).
- **Прод-сборки:** по общему ТЗ §3 в прод по умолчанию ожидается реальный транспорт, когда будет готов; на B3 допускается техническая заглушка при выключенном моке, без обхода явного выбора пользователя/конфига для мока в dev.

## Критерии приёмки

1. Описан и реализован **единый контракт** взаимодействия с контроллером (исходящие команды + доставка входящих ответов/событий подписчикам), достаточный для мока и для будущей подстановки реального транспорта без переписывания потребителей.
2. Реализована **мок-реализация**, по смыслу покрывающая эталон `MockControllerConnection`: запись исходящих команд (тип команды + тело/байты); метод или эквивалент для **имитации входящего ответа** с полезной нагрузкой.
3. Реализована **реализация-заглушка** «реального» транспорта (до B1), удовлетворяющая контракту и не ломающая приложение при выключенном моке.
4. **Hilt** (или согласованный с проектом механизм) подставляет мок или заглушку в зависимости от сохранённого значения **`USE_MOCK_CONTROLLER`** / того же ключа, что использует экран сервиса.
5. На эмуляторе выполняется проверка: **тестовая команда** через шлюз → в логе и/или UI виден **ожидаемый результат** (запись команды и/или обработка симулированного ответа).
6. Добавлены **автотесты** на уровне, согласованном с проектом (юнит-тесты на мок: запись команд, симуляция ответа), если в репозитории уже принят минимальный бар для нового кода; иначе приёмка по п.5 остаётся обязательной, тесты — по возможности без блокировки этапа.
7. В каталоге модуля complex присутствует **`summary.md`** с кратким итогом и указанием, как воспроизвести сценарий п.5 (DoD общего ТЗ для B3).
