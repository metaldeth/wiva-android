# Ревью B3: шлюз контроллера (mock-first)

**Область:** `hardware/controller/*`, `di/ControllerModule.kt`, `ServiceViewModel.kt`, `ServiceScreen.kt`.

## Краткое резюме

Реализация соответствует заявленной архитектуре stage-b-mock-first: единый контракт `ControllerGateway`, `@Singleton` `DelegatingControllerGateway` с чтением `USE_MOCK_CONTROLLER` на каждом исходящем вызове, мок с журналом TX и эмуляцией RX, заглушка «реала» с логированием, Hilt через `@Binds` на делегат. Поток ответов собран как `merge(mock, stub).shareIn(@AppIoScope, Eagerly)` — подписчики видят события из обеих веток без смены подписки при переключении флага; активная ветка определяется только при `sendCommand` / `simulateResponseForTests`, что согласовано с ТЗ. Перечисления `RequestCommand` / `ResponseCommand` с `fromCode` покрывают контракт протокола. На экране сервиса добавлены переключатель (уже был) и кнопка UC-3 с вызовом шлюза + `simulateResponseForTests`. Юнит-тесты `MockControllerGatewayTest` проверяют запись команд и эмиссию в flow.

**Вердикт:** для границ задачи B3 код пригоден к интеграции; критических дефектов, блокирующих слияние по функционалу B3, не выявлено.

## Блокирующие замечания

Нет.

*(Под «блокирующими» здесь имеются в виду ошибки, из-за которых нельзя принимать задачу: неверная маршрутизация мок/реал, утечки/краши DI, несовместимость с ключом конфига, поломка сборки — таких пунктов нет.)*

## Рекомендации (неблокирующие)

1. **`runControllerSelfTest` и `delay(150)`** — хрупкая синхронизация; для демо UC-3 допустимо, но при появлении подписчиков на `incomingResponses` лучше убрать магическую задержку (например, колбэк после записи в мок, или явная последовательность в тесте/UI без предположения о тайминге).

2. **`ControllerResponseEvent` и `SentCommandRecord` как `data class` с `ByteArray`** — `equals`/`hashCode` по ссылке на массив, не по содержимому. Для событий в Flow обычно не мешает; если когда-нибудь появятся дедупликация или кэширование по событию — стоит заменить на обёртку с content-based equals или не использовать эти типы в `Set`/`distinctUntilChanged` без осторожности.

3. **UX самопроверки** — при выключенном моке `simulateResponseForTests` — no-op: кнопка «Тест контроллера» всё равно шлёт TX в стаб, но RX не симулируется; пользователю это не объяснено. Имеет смысл подписать подсказку в UI или показывать Snackbar/лог только в мок-режиме.

4. **`observeUpdateProgress` / прочие `catch (e: Exception)`** — если в блок попадает `CancellationException` (наследник `Exception` в стандартной иерархии Kotlin), отмена корутины может не проброситься. Паттерн: пробрасывать `CancellationException` или не оборачивать весь `collect` широким `catch`. Часть этого кода затрагивает не только B3, но при следующем касании файла стоит выровнять с best practices coroutines.

5. **`StubRealControllerGateway._incoming`** — сейчас не используется для эмиссии; это запас под будущий транспорт. Ок для заглушки; можно кратко зафиксировать в KDoc, чтобы не казалось «мёртвым кодом».

6. **Согласованность тега логов** — `MockControllerGateway` / `StubRealControllerGateway` и вызов из `ServiceViewModel` используют `"WivaController"`; единообразно, фильтрация в logcat удобна.
