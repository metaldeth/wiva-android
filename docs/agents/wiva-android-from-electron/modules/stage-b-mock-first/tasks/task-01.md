# task-01: B3 — MockController, делегирующий шлюз, Hilt

**Зависимости:** —

## Цель

Реализовать задачу **B3** из ТЗ модуля `stage-b-mock-first`: единый контракт доступа к контроллеру, мок по смыслу `MockControllerConnection`, заглушка «реального» транспорта, переключение **в рантайме** по значению `JsonStoreKeys.USE_MOCK_CONTROLLER` из `ConfigRepository`, проверочный сценарий на эмуляторе и артефакт `summary.md` в каталоге модуля complex.

## Контекст

- Переключатель «Мок контроллера» уже пишет в Room JsonStore через `ServiceViewModel.setUseMockController` и ключ `JsonStoreKeys.USE_MOCK_CONTROLLER` (`useMockController`).
- Потребители в будущем (B4 и далее) должны зависеть только от **шлюза**, а не от мока/serial напрямую.

## Файлы: создать (ориентир по пакету)

Пакет согласовать с текущей структурой `app` (рекомендация: `com.viwa.android.hardware.controller` или `com.viwa.android.data.controller` — единообразно с остальным приложением).

| Файл / область | Назначение |
| -------------- | ---------- |
| `RequestCommand.kt` (или эквивалент) | Исходящие команды: значения **1:1** с эталоном wiva (см. ссылку ниже). |
| `ResponseCommand.kt` (или эквивалент) | Входящие ответы: значения **1:1** с эталоном wiva. |
| Интерфейс шлюза (например `ControllerGateway`) | Исходящая отправка + API подписки на входящие события (Flow / callback — по стилю проекта). |
| `MockControllerGateway` (имя по согласованию) | Журнал отправленных команд; метод симуляции входящего ответа; доставка подписчикам как у реального потока. Опционально: флаг в духе `isMockPort` как точка расширения для будущих наливов, если use case ещё нет — без жёсткой интеграции. |
| `StubRealControllerGateway` (или аналог) | Заглушка: лог «не подключено» / no-op, без зависимостей от B1/B2. |
| `DelegatingControllerGateway` | `@Singleton`: на каждом вызове читает `configRepository.get(JsonStoreKeys.USE_MOCK_CONTROLLER) == "true"` и делегирует моку или заглушке; подписки на события маршрутизировать согласованно (один источник событий активной ветки или переключение подписок — без утечек и с учётом Main). |
| Hilt-модуль (новый или дополнение к `AppModule` / отдельный `ControllerModule`) | `@Binds` / `@Provides`: снаружи внедряется **только** интерфейс шлюза → реализация делегата; мок и stub — доступны делегату. |
| `docs/.../stage-b-mock-first/summary.md` | Итог этапа + шаги воспроизведения сценария приёмки п.5 ТЗ. |

Значения исходящих и входящих команд брать из эталона **wiva_electron** (не выдумывать коды):

## Критерии приёмки

1. Описан и реализован единый контракт (исходящие + входящие для подписчиков), достаточный для подстановки реального транспорта позже без смены API потребителей шлюза.
2. Мок фиксирует исходящие команды (тип + полезная нагрузка байтами) и умеет **симулировать** входящий ответ, получаемый подписчиками тем же механизмом, что и «живой» поток.
3. Заглушка реала реализует контракт и не падает при выключенном моке (эмулятор без железа — ожидаемо только логирование/no-op).
4. Hilt отдаёт в код приложения делегирующий шлюз; выбор мок/реал определяется **актуальным** значением `USE_MOCK_CONTROLLER` на момент вызова (UC-1, UC-2, переключение без перезапуска процесса).
5. На эмуляторе: тестовая команда через шлюз → в logcat и/или UI видно ожидаемый результат (отправка и/или обработка симулированного ответа).
6. Автотесты: юнит-тесты на мок (запись команд, симуляция ответа) — **желательно**, если в проекте уже есть бар; при `tdd: false` в `AGENTS.md` отсутствие unit-тестов не отменяет п.5.
7. `summary.md` в `docs/agents/viwa-android-from-electron/modules/stage-b-mock-first/` описывает, как воспроизвести п.5.
8. `gradlew.bat assembleDebug` (Windows) / `./gradlew assembleDebug` завершается успешно.

## НФТ (напоминание)

- Потокобезопасность журнала мока и диспетчеризации событий; UI — только с Main при необходимости.
- Не тянуть USB/serial и полный парсер протокола (B1/B2).
- Логи с понятными тегами: исходящая команда в мок, симулированный ответ.
