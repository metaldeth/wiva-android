# Ревью ТЗ: stage-b-mock-first, B3 (MockController)

> Проверка по чеклисту tz-reviewer относительно постановки из `request.md`.

## Краткий вывод

ТЗ в целом полно отражает задачу B3 из `request.md` и строку B3 таблицы модуля B в `TZ_WIVA_ANDROID_FROM_ELECTRON.md`: контракт, мок по смыслу `MockControllerConnection`, заглушка «реала», Hilt и ключ `USE_MOCK_CONTROLLER`, юзер-кейсы, НФТ и критерии приёмки согласованы с границами этапа (вне скоупа B1/B2/B4). Для `wiva-android` в `AGENTS.md` нет `browserTesting: true` — отдельный раздел «UI-сценарии для тестирования» не требуется по правилам ревьюера. Перед передачей в реализацию стоит снять противоречие между UC-1 и абзацем о смене флага во время работы приложения.

## Критичные проблемы

1. **UC-1 и переключение флага во время сессии** — **закрыто:** в `tz.md` зафиксирована стратегия «шлюз читает `USE_MOCK_CONTROLLER` на каждой операции и делегирует» (без обязательного cold start). Исходное противоречие с вариантом «только со следующего запуска» снято.

## Некритичные замечания

- **§3 общего ТЗ** описывает полную проверку «выбор напитка → оплата (мок) → команда наливу»; модульное ТЗ корректно сужает DoD до тестовой команды по строке B3. Имеет смысл одной фразой в ТЗ отослать читателя к отложенному сценарию (B4 и последующие задачи), чтобы при сверке с §3 не возникало впечатления пробела.
- **«Каталог модуля complex»** в критерии 7 можно заменить на явный путь к `wiva-android/docs/agents/wiva-android-from-electron/modules/stage-b-mock-first/`, чтобы не путать с термином complex-режима Cursor.
- **UC-3:** формулировка «согласованная тестовая точка» осознанно гибкая; для снижения риска срыва приёмки п.5 можно в `summary.md` (как уже предложено) зафиксировать конкретное место вызова (dev-кнопка, тест, временный хук) — в ТЗ это можно упомянить как пожелание, не как новый обязательный артефакт.
- **Критерий 6 (автотесты):** условная формулировка адекватна; при появлении минимального бара в проекте стоит заранее согласовать, что именно покрывается юнит-тестами мока (журнал команд + симуляция ответа), чтобы не спорить на границе этапа.
