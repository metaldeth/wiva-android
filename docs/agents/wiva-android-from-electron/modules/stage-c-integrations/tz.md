# ТЗ: этап stage-c-integrations (модуль C — MAX, СБП, Нанокасса)

> Источник границ этапа: `wiva-android/docs/TZ_WIVA_ANDROID_FROM_ELECTRON.md` (модуль C, §2, §4.1). Запрос: `request.md` в этом каталоге.

## Описание задачи

Перенести в `wiva-android` интеграции **MAX**, **СБП** и **Нанокасса** как **слой данных и домена** (клиенты удалённых API, репозитории, настройки, при необходимости UI-паттерны из киоска), с минимальной адаптацией под пакет и структуру приложения wiva. **Источник реализации — только `legacy Android kiosk`**, в соответствии с **§2** основного ТЗ: области MAX, СБП, Нанокасса брать из эталонных путей киоска (`data/remote/max`, `data/remote/sbp`, `data/remote/nanokassa`, связанные `domain/repository`, реализации вроде `NanoKassaRepositoryImpl` / `MaxRepository` и смежный код). Самовольная замена SDK или иной стек без согласования с владельцем продукта запрещены.

Этап выполняется **волнами C1 → C2 → C3 → C4**: после **C1**, **C2** и **C3** обязательна **успешная промежуточная сборка**; **нельзя** объединять C1–C3 в один PR без таких промежуточных сборок. Завершение этапа: рабочая сборка по `buildCommand` из `wiva-android/AGENTS.md` (`gradlew.bat assembleDebug` на Windows), артефакт `summary.md` в каталоге модуля.

Семантика обменов с телеметрией и сценарии оплаты «как у автомата wiva» на этом этапе **не дублируются** — они в модуле D и далее; здесь задача — **встроить готовые киосковые интеграции** в кодовую базу wiva-android и обеспечить DI и точки конфигурации.

## Юзер-кейсы и сценарии

### Актёры

- **Разработчик / инженер** — переносит код, проверяет сборку, настройки, тесты.
- **Оператор автомата (косвенно)** — в перспективе использует сервисные настройки MAX/СБП/НК; на этапе C достаточно проверки через заглушечный/сервисный экран или эквивалент, как в DoD основного ТЗ для C1.

### Основные сценарии

1. **C1 — MAX**  
   После переноса слоя данных и domain для MAX приложение **собирается**. Есть способ **вызвать настройки MAX** (как в сервис-меню киоска) с **заглушечного или минимального экрана** — без полного UI автомата wiva, если он ещё не готов.

2. **C2 — СБП**  
   После переноса репозиториев, парсеров и при необходимости use case для СБП приложение **снова собирается**. **Юнит-тесты парсеров** (как минимум smoke) проходят.

3. **C3 — Нанокасса**  
   После переноса слоя Нанокассы приложение **снова собирается**; проверки аналогичны по смыслу C1/C2 (сборка + согласованный с архитектурой минимальный способ убедиться, что модуль подключён).

4. **C4 — DI и конфигурация**  
   MAX, СБП и Нанокасса подключены через **единую модель DI** (Hilt или как принято в эталоне киоска), с **едиными точками конфигурации** (базовые URL, ключи, флаги — по образцу киоска, адаптированные под wiva). Проверка: **один интеграционный тест** **или** ручная проверка, что зависимости для этих интеграций **корректно инжектируются** без дублирования конфликтующих модулей.

### Альтернативные и отрицательные сценарии

- При **неоднозначности** поведения или приоритета, отсутствующей в ТЗ и эталонных репозиториях — **остановка и вопрос владельцу** по правилам **§4.1** основного ТЗ (контекст, противоречие, варианты для бизнеса, один функциональный вопрос).
- Если эталон киоска и ожидания wiva **расходятся** по смыслу оплаты/телеметрии — на этапе C **приоритет источника для интеграций остаётся киоск**; противоречия с семантикой wiva фиксируются для модуля D, **не** «угадываются» в коде C без ADR/владельца.

### Связь с моком контроллера (§3)

Где сценарии оплаты завязаны на контроллер, для эмулятора используется **мок контроллера**; перенос интеграций C не отменяет требование мока, но **полная сквозная оплата** может завершаться на последующих этапах. На C достаточно DoD по таблице модуля C основного ТЗ.

## Нефункциональные требования (если есть)

- **Сборка:** после каждой из волн C1, C2, C3 — успешный запуск `assembleDebug` (см. `AGENTS.md`).
- **Соответствие эталону:** структура и зависимости интеграций — по **legacy Android kiosk**, без произвольных альтернативных SDK (§2).
- **Изоляция PR:** C1–C3 не сливать в один PR без промежуточных зелёных сборок (явное ограничение из `request.md`).
- **Тестирование:** для C2 — минимум smoke по парсерам; для C4 — интеграционный тест инъекций или согласованная ручная проверка.
- **Документация этапа:** по завершении — `summary.md` в `docs/agents/wiva-android-from-electron/modules/stage-c-integrations/`.

## Критерии приёмки

| ID | Критерий |
|----|----------|
| C1 | Перенесены data/domain для MAX из **legacy Android kiosk** (§2); проект **собирается**; настройки MAX **доступны** с заглушечного экрана по аналогии с сервис-меню киоска. |
| C2 | Перенесён слой СБП из **legacy Android kiosk**; **сборка OK**; **юнит-тесты парсеров** (минимум smoke) проходят. |
| C3 | Перенесена Нанокасса из **legacy Android kiosk**; **сборка OK**; проверки по смыслу согласованы с C1/C2 (без ослабления требования §2). |
| C4 | DI и единые точки конфигурации для MAX, СБП, НК введены; **интеграционный тест или ручная проверка инъекций** пройдены. |
| Общий | Нет объединения C1–C3 в одном PR без промежуточных успешных сборок; при блокирующих неясностях зафиксированы вопросы по §4.1, а не «додуманное» поведение. |
