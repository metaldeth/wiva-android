# task-f1 — Репозиторий: чеклист F1 + инструкция мок↔реал (опционально AGENTS.md)

**Зависимости:** —

## Привязка к ТЗ / приёмке

- **F1-R1:** шаблон чеклиста офисной приёмки — контроллер, порт/скорость со ссылкой на `SERIAL_PORTS_IMPLEMENTATION.md` и §2 основного ТЗ; сценарии регрессии **против мока** (шаги согласованы с §3 и сценарием эмулятора).
- **F1-R2:** в шаблоне — поля **исполнитель, дата, подпись**, версия APK/build number, идентификатор стенда (заполнение на стенде).
- **Репо (общие):** инструкция по портам и переключению **мок ↔ реал** со ссылками на §2–§3 основного ТЗ (`wiva-android/docs/TZ_WIVA_ANDROID_FROM_ELECTRON.md` или актуальный путь).
- **Онсайт F1-O1…O3** — не входят в эту задачу; в чеклисте только **строки-шаблоны** для отметок и комментариев.

## Конкретные шаги

1. **Primary-путь файла** (зафиксировано в `plan.md`): **`wiva-android/docs/OFFICE_HARDWARE_CHECKLIST.md`** — единый шаблон F1+F2.
2. **Создать шаблон чеклиста** с разделами:
   - метаданные стенда: APK/build, идентификатор стенда, исполнитель, дата, подпись (пустые поля/плейсхолдеры);
   - **контроллер:** подключение, проверка порта и скорости — явная ссылка на `wiva_electron/docs/SERIAL_PORTS_IMPLEMENTATION.md` и на §2 основного ТЗ;
   - **мок ↔ реал:** краткая инструкция: флаг `JsonStoreKeys.USE_MOCK_CONTROLLER` / сервисное меню; для стенда F1 — **мок выключён** (как в ТЗ); ссылки на `ControllerPortSettings.kt`, `ControllerModule.kt`, `DelegatingControllerGateway` / эталонные пути из `architecture.md`;
   - **регрессия vs мок:** таблица или нумерованный список шагов (минимум happy-path: выбор напитка → оплата / согласование с F2 или мок оплаты → команда наливу / ожидаемые ответы); колонки «ожидание (мок/electron)» / «факт на стенде» / «заметки»;
   - **заметка по ограничению:** если реальный байтовый обмен на Android ещё невозможен (заглушка транспорта) — не скрывать; оформить как блок «блокер стенда / отдельная задача по §4.1» (как в `architecture.md`).
3. **Опционально — дубль/указатель** в `docs/agents/wiva-android-from-electron/modules/stage-f-office/`: короткий `.md` со ссылкой на primary-файл в `wiva-android/docs/`, если команда держит агентский трек отдельно.
4. **Опционально — `wiva-android/AGENTS.md`:** добавить компактный подраздел (например «Офисный стенд / этап F») со ссылками на шаблон чеклиста, основное ТЗ §3, `SERIAL_PORTS_IMPLEMENTATION.md` в wiva_electron — без дублирования длинных инструкций.
5. **`summary.md`** в каталоге `.../modules/stage-f-office/`: создать или обновить структуру по `architecture.md` — раздел **«Сделано в репозитории»** с отметками F1-R1, F1-R2 и ссылкой на файл шаблона; раздел **«Подтверждено на стенде»** оставить с плейсхолдерами до онсайта; **«Статус модуля F»** — «готово к стенду» / «ожидает стенд», если онсайт не выполнялся.
6. **Ревью:** проверить отсутствие PAN/CVV и лишних платёжных данных в шаблоне; соответствие критериям таблицы приёмки ТЗ.
7. **Сборка:** выполнить `buildCommand` из `wiva-android/AGENTS.md` перед финализацией PR (регрессия проекта).

## Не делать в рамках F1

- Реализацию USB/serial без отдельного решения после стенда.
- Заполнение онсайт-строк и подпись (инженер стенда).
