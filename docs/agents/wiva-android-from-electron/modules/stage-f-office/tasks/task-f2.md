# task-f2 — Репозиторий: раздел «терминал» в чеклисте (шаблон F2-R1)

**Зависимости:** —  
*(Практически: тот же файл чеклиста, что и в task-f1; при параллели — одна ветка либо rebase/merge после появления файла из task-f1.)*

## Привязка к ТЗ / приёмке

- **F2-R1 (репо):** в **том же** чеклисте (или явно связанном документе, если команда разнесёт) — подпункт **«терминал»**: проверка связи, тип терминала, ссылки на эталон **wiva_electron** — `PaymentTerminalService.ts`, `src/main/modules/payment/` (`index.ts`, `service.ts`, `manager.ts`, `preparing/`, `paymentTypes/`).
- **F2-O1, F2-O2** — **только онсайт:** успешная транзакция, согласованность цепочки оплата → контроллер; в шаблоне — **поля для фиксации** (время, идентификатор сессии/чека в допустимом для аудита объёме, **без** PAN/CVV и полных реквизитов карты), чекбоксы/строки «выполнено», подпись — как продолжение F1-R2.

## Конкретные шаги

1. **Открыть primary-файл шаблона чеклиста** (созданный в task-f1 в `wiva-android/docs/`). Если task-f1 ещё не смержен — работать в общей ветке с автором F1 или добавить раздел отдельным коммитом поверх его ветки.
2. **Добавить раздел «Платёжный терминал (F2)»** (или эквивалентное название):
   - настройка стенда: терминал подключён и сконфигурирован по правилам стенда; отсылка к **wiva_electron** для целевого типа терминала (пути выше);
   - проверка связи приложения с терминалом (чеклистные пункты без привязки к конкретному драйверу Android — «по регламенту стенда / документации electron»);
   - поле **тип терминала** (текст/выбор);
   - сценарий: **полная успешная транзакция** (тестовая или боевая среда — пометка в шаблоне «по регламенту Wiva»);
   - цепочка **оплата → события → команды контроллеру** — строки для отметки согласованности с wiva (в том же прогоне или сразу после);
   - блок **фиксации для аудита стенда** (внутренние правила): дата/время, ID сессии/чека — плейсхолдеры; напоминание о PCI/внутренних ограничениях.
3. **Связать с F1:** в happy-path регрессии (раздел task-f1) при необходимости одна фраза-перекрёстная ссылка: «оплата в этом прогоне — реальный терминал (F2) или мок оплаты по договорённости стенда».
4. **Обновить `summary.md`** модуля: в «Сделано в репозитории» отметить F2-R1 и ссылку на тот же файл чеклиста; онсайт F2-O* не отмечать до стенда.
5. **Ревью:** согласованность формулировок с ТЗ F2; отсутствие чувствительных данных в примерах.
6. **Сборка:** всегда выполнить `buildCommand` из `wiva-android/AGENTS.md` перед финализацией PR при любых изменениях под `wiva-android/` (как в task-f1, единообразие с NFR ТЗ).

## Онсайт (вне задачи исполнителя репо)

- Закрытие **F2-O1, F2-O2** — **sign-off** инженером стенда: заполнение полей, подпись и дата (как в F1-R2), хранение экземпляра по политике команды и ссылка из `summary.md`.
